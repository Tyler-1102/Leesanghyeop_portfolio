## 1. 문제 발견

6명으로 구성된 개발 프로젝트 팀에 신규 팀원 1명이 합류했습니다. 신규 팀원은 **Jira에 업무 항목을 등록하는 순서**와 팀이 주로 사용하는 **GitHub Desktop 기반의 브랜치·머지 절차**를 익히는 과정에서 어려움을 겪었습니다.

그때마다 주변 팀원에게 다음 행동을 확인해야 했고, 일부 업무가 간헐적으로 늦어졌습니다. 저는 이를 개인의 숙련도 문제가 아니라 **신규 구성원이 팀의 일하는 방식을 한 번에 파악할 수 있는 온보딩 체계가 없는 문제**로 정의했습니다.

## 2. 원인 파악

신규 팀원의 옆에서 실제 업무를 함께 수행하며 질문이 발생하는 지점을 확인했습니다. 혼선은 특정 도구의 기능보다 여러 도구를 오가는 **전체 업무 순서와 팀 내부 기준**에서 주로 발생했습니다.

- Jira 업무 항목과 LabSSAFY 이슈를 어떤 순서로 만드는가
- 이슈 번호에 맞춰 브랜치를 어떻게 생성하는가
- GitHub Desktop에서 어느 브랜치를 기준으로 분기하는가
- 작업 완료 후 MR을 만들고 리뷰를 반영한 다음 Jira를 언제 완료 처리하는가
- 스토리 포인트, Git 라벨, 브랜치명, 커밋 메시지에 어떤 기준을 적용하는가

기존 규칙은 팀원들의 경험과 구두 설명에 흩어져 있었습니다. 한 번의 설명만으로는 업무 중 다시 참고하기 어렵기 때문에, **실제 작업 순서대로 찾아볼 수 있는 문서**가 필요하다고 제안했습니다.

## 3. 목표 설정

온보딩의 목표를 단순히 “문서를 만든다”가 아니라 다음과 같이 잡았습니다.

1. 신규 팀원이 현재 단계에서 해야 할 다음 행동을 스스로 찾을 수 있게 한다.
2. Jira·Git·MR을 오갈 때 팀마다 다른 업무 기준을 하나의 흐름으로 정리한다.
3. 구두 교육이 끝난 뒤에도 같은 내용을 반복해서 확인할 수 있게 한다.

다만 당시 목표 달성 여부를 판단할 **질문 횟수·소요 시간 등의 사전 지표는 설정하지 못했습니다.** 이 부분은 결과를 과장하지 않고 후속 개선 과제로 남겼습니다.

## 4. 실행 과정

### 1:1 실무 교육

신규 팀원의 옆에서 Jira와 GitHub Desktop을 직접 사용하며 업무 흐름을 안내했습니다. 기능을 나열하기보다 실제 업무 하나가 생성되고 완료되는 순서에 맞춰 설명했습니다.

### 교육 후 매뉴얼 재작성

교육 과정에서 실제로 나온 질문과 막힌 지점을 반영해 매뉴얼을 다시 작성했습니다. 문서는 다음 다섯 영역으로 구조화했습니다.

1. **전체 워크플로우** — Jira 업무 생성 → LabSSAFY 이슈 → 브랜치 생성 → 작업 및 MR → 리뷰 반영 → 머지 → Jira 완료
2. **업무 난이도 기준** — 스토리 포인트를 작업 단위에만 부여하고 1·2·3·5·8점으로 판단하는 기준
3. **변경 유형 기준** — feat, fix, docs, refactor, test 등 Git 라벨의 의미
4. **브랜치·커밋 기준** — develop 기준 분기, 이슈 번호를 포함한 브랜치명, 커밋 제목과 설명 작성법
5. **실수 예방 체크포인트** — 새 브랜치 생성 전 fetch, MR의 closes 이슈 번호 입력 등

### 반복 확인이 가능한 채널에 공유

완성한 문서는 팀의 주요 소통 채널인 Discord 공지에 게시했습니다. 질문이 생겼을 때 구두 설명을 기다리지 않고 다시 확인할 수 있도록 하기 위한 선택이었습니다.

## 5. 결과와 검증 범위

### 확인된 결과

- 실제 적용 대상은 **신규 팀원 1명**이었습니다.
- 해당 팀원이 매뉴얼을 사용했습니다.
- 사용자는 **업무 수행 방법을 구체적으로 알 수 있어 좋았다**는 취지의 피드백을 전달했습니다.

### 아직 확인되지 않은 결과

아래 항목은 당시 별도로 기록하지 않아 현재 공란으로 남겨두었습니다.

| 검증 지표 | 결과 |
| --- | --- |
| 문서 공유 전후 반복 질문 횟수 | — |
| 첫 이슈를 독립적으로 처리했는지 | — |
| 업무 시작까지 걸린 시간의 전후 변화 | — |
| 문서 재열람 또는 활용 횟수 | — |

따라서 이 경험을 **“팀 전체 생산성을 향상했다”거나 “업무 지연을 해소했다”라고 확대해 표현하지 않았습니다.** 확인할 수 있는 범위는 신규 팀원 1명의 온보딩을 지원하고, 구두로 흩어진 업무 기준을 재사용 가능한 문서로 전환했다는 점까지입니다.

## 6. 한계와 후속 개선안

문서의 사용이 확인된 사람은 신규 팀원 1명이었고, 기존 팀원의 활용 여부와 사용하지 않은 이유는 확인하지 못했습니다. 기존 구성원에게 이미 익숙한 내용이었는지, Discord 공지에서 문서를 다시 찾기 어려웠는지 구분해서 조사하지 못한 점이 한계입니다. 따라서 문서를 만들었다는 사실만으로 지속적인 활용을 보장할 수 없었습니다.

다시 설계한다면 다음과 같이 보완하겠습니다.

- 합류 첫날 확인하는 **온보딩 체크리스트**에 문서 링크 포함
- 첫 업무 전 **15분 안내 세션** 운영
- 자주 막히는 단계만 모은 **FAQ와 화면 예시** 추가
- 온보딩 전후 질문 횟수와 첫 업무 독립 수행 시간을 기록
- 1주 후 짧은 인터뷰를 통해 이해도와 추가 불편 확인

## 7. HR 직무와의 연결

이 경험을 통해 온보딩은 정보를 많이 전달하는 일이 아니라, **신규 구성원이 실제 업무를 수행하는 과정에서 어디서 막히는지 관찰하고 필요한 기준을 적절한 시점에 제공하는 일**임을 배웠습니다.

- **온보딩·교육 운영** — 신규 구성원의 혼선을 관찰하고 실제 업무를 바탕으로 1:1 교육
- **프로세스 개선** — 구두로 흩어진 업무 규칙을 처음부터 끝까지 이어지는 절차로 표준화
- **문서 커뮤니케이션** — 실무자가 바로 따라 할 수 있도록 규칙, 예시, 주의사항을 함께 구성
- **피드백 기반 개선** — 교육에서 나온 질문을 매뉴얼 재작성에 반영
- **성과 검증 태도** — 확인되지 않은 효과를 과장하지 않고 다음 운영에서 측정할 지표를 정의

작은 팀의 한 명을 대상으로 한 경험이지만, 구성원의 적응 문제를 개인의 부족함으로 보지 않고 **업무 환경과 정보 구조의 문제로 바라본 것**이 이 경험에서 얻은 가장 중요한 HR 관점입니다.
